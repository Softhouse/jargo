Class ArgumentBuilder<SELF extends ArgumentBuilder<SELF,T>,T>

  • Type Parameters:
    SELF - the type of the subclass extending this class. Concept borrowed from: Ansgar.Konermann's blog The pattern also resembles the Curiously recurring template pattern
    T - the type of arguments the built Argument instance should handle, such as Integer in the case of Arguments.integerArgument(String...)

    ArgumentBuilder.ArityArgumentBuilder, ArgumentBuilder.CommandBuilder, ArgumentBuilder.DefaultArgumentBuilder, ArgumentBuilder.MapArgumentBuilder, ArgumentBuilder.OptionArgumentBuilder, ArgumentBuilder.RepeatedArgumentBuilder, ArgumentBuilder.SplitterArgumentBuilder

    @NotThreadSafe
public abstract class ArgumentBuilder<SELF extends ArgumentBuilder<SELF,T>,T>
extends Object
     Responsible for configuring and building Argument instances.
 Example builders can be created with Arguments.
 
 Note:The code examples (for each method) assumes that all methods in Arguments have been statically imported.
 
 Note:Some methods needs to be called in a specific order
 (to make the generic type system produce the correct type) and to guide the
 caller, such invalid orders are documented with Deprecated. If those warnings
 are ignored IllegalStateException will be thrown at the offending call.

         Parses command line arguments and returns the value of the argument built
 by build(), providing a simple one liner shortcut when faced with only one argument.

      • metaDescription

        public final SELF metaDescription(String aMetaDescription)
         By default StringParsers provides a meta description (by implementing StringParser.metaDescription()
 that describes the type of data they expect. For instance, if you're writing a music player,
 a user of your application would (most probably) rather see:
 
 --track-nr <track nr>    The track number to play (using metaDescription("<track nr>") )
 
 instead of
 
 --track-nr <integer>     The track number to play (the default provided by StringParser.metaDescription())
 
 So when using general data type parsers such as StringParsers.integerParser() you're better of if
 you provide a meta description that explains what the integer represents in the context of this argument.
 
 Note: empty meta descriptions aren't allowed
 Note: the surrounding < & > aren't enforced or added automatically but it's preferred
 to have them because it makes a clear distinction between argument names and their parameters.
        Parameters:
        aMetaDescription - "<track nr>" in the above example
        Returns:
        this builder
        Throws:
        IllegalArgumentException - if aMetaDescription is empty

        @Beta
public SELF limitTo(Predicate<? super T> aLimiter)
         Limits values parsed so that they conform to some specific rule.
 Only values for which Predicate.apply(Object) returns true, will be accepted.
 Other values will cause an ArgumentException.
 For example Range.closed(Comparable, Comparable) only allows values within a range.
 
 To override the default error message that is generated with Texts.UserErrors.DISALLOWED_VALUE
 you can throw IllegalArgumentException from Predicate.apply(Object). The detail
 message of that exception will be used by ArgumentException.getMessageAndUsage().
 When this is needed it's generally recommended to write a parser of its own instead.
 
 Note:toString() on aLimiter will replace StringParser.descriptionOfValidValues(Locale) in the usage
 
 Note:The validity of any default value isn't checked until
 it's actually needed when ParsedArguments.get(Argument) is called. This is so
 because Supplier.get() (or StringParser.defaultValue()) could take an arbitrary long time.
        Parameters:
        aLimiter - a limiter
        Returns:
        this builder

        @CheckReturnValue
public final <K> ArgumentBuilder.MapArgumentBuilder<K,T> asKeyValuesWithKeyParser(StringParser<K> keyParser)
         Makes this argument handle properties like arguments:
 -Dproperty_name=value
 where "-D" is one of the strings supplied to names(String...),
 "property_name" is decoded by keyParser and
 "value" is decoded by the StringParser previously passed to the constructor.

        For this to work correctly it's paramount that K implements a proper Object.hashCode() because it's going to be used a key in a Map.

      • variableArity

        @CheckReturnValue
public ArgumentBuilder.ArityArgumentBuilder<T> variableArity()
        Useful for handling a variable amount of parameters in the end of a command. Uses this argument to parse values but assumes that all the following parameters are of the same type, integer in the following example:

 List<Integer> numbers = integerArgument("--numbers").variableArity().parse(threeArgs);

      • repeated

        @CheckReturnValue
public ArgumentBuilder.RepeatedArgumentBuilder<T> repeated()
         Makes it possible to enter several values for the same argument. Such as this:
 
 
         
 String[] arguments = {"--number", "1", "--number", "2"};
 List<Integer> numbers = integerArgument("--number").repeated().parse(arguments);
 assertThat(numbers).isEqualTo(asList(1, 2));
 
 
        
 
 If you want to combine repeated() with a specific arity(int) then call
 arity(int) before calling repeated():
 
 
         
 String[] arguments = {"--numbers", "1", "2", "--numbers", "3", "4"};
 List<List<Integer>> numberLists = integerArgument("--numbers").arity(2).repeated().parse(arguments);
 assertThat(numberLists).isEqualTo(asList(asList(1, 2), asList(3, 4)));
 
 
        
 
 For repeated values in a property map such as this:
 
 
         
 String[] arguments = {"-Nnumber=1", "-Nnumber=2"};
 Map<String, List<Integer>> numberMap = integerArgument("-N").repeated().asPropertyMap().parse(arguments);
 assertThat(numberMap.get("number")).isEqualTo(asList(1, 2));
 
 
        
 
 repeated() should be called before asPropertyMap().
 For arguments without a name (indexed arguments) use variableArity() instead.
        Returns:
        a new (more specific) builder
