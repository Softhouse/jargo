public abstract class Command

 Commands automatically gets an invocation of execute when given on the command line.
 This is particularly useful to avoid a never ending switch statement.
 
 Commands have a CommandLineParser themselves (and thereby sub-commands are allowed as well), that is,
 they execute a command and may support contextual arguments as specified by the constructor Command(Argument...).
 
 Sub-commands are executed before their parent Command.
 
 To integrate your Command into an Argument use Arguments.command(Command)
 or CommandLineParser.withCommands(Command...) if you have several commands.
 
 If you support several commands and a user enters several of them at the same
 time they will be executed in the order given to CommandLineParser.parse(String...).
 If any parse errors occurs (for command arguments) the Command
 will not be executed. However, if given multiple commands and parse errors occurs,
 all Commands given before the Command with parse errors will have been executed.
 This is so because command arguments are allowed to be dependent on earlier Commands being executed.
 So it's recommended to let the user know when you've executed a Command.
 
 Mutability note: although a Command should be Immutable
 the objects it handles doesn't have to be. So repeated invocations of execute
 is allowed to yield different results or to affect external state.
Example subclass: 
 
 public class Build extends Command
 {
   private static final Argument<File> PATH = Arguments.fileArgument().description("the directory to build").build();
 
   public Build()
   {
     super(PATH);
   }
 
   public String description()
   {
     return "Builds a target";
   }
 
   protected void execute(ParsedArguments parsedArguments)
   {
     File pathToBuild = parsedArguments.get(PATH);
     //Build pathToBuild here
   }
 }
And the glue needed to integrate the Build Command with a CommandLineParser: 
 
 CommandLineParser.withCommands(new Build()).parse("build", "some_directory_that_needs_building");
As can be seen in the example the command name is the class name in lower case by default. This may be overridden with commandName().
